## 서비스 개요

"오늘 뭐 먹지?"는 매일 반복되는 메뉴 결정 피로를 해소하는 랜덤 메뉴 추천 웹 서비스다.
식약처 공식 레시피(한식/분식)와 TheMealDB API(중식/일식/양식)를 결합해
카테고리별로 다양한 메뉴를 슬롯머신 방식으로 추천하고, 재료·레시피·영양정보·블로그 후기까지
한 화면에서 제공한다.

- 기술 스택: React 18 + TypeScript + Vite + Tailwind CSS
- 배포: Cloudflare Pages (서버리스, DB 없음)
- 외부 의존: 식약처 API, TheMealDB API, Pexels API, 네이버 블로그 검색 API, 카카오 공유 SDK

---

## 핵심 문제 (Why)

| 문제 | 현실 상황 |
|------|-----------|
| 메뉴 결정 피로 | "뭐 먹지?"는 매일 평균 2~3회 이상 발생하는 의사결정. 선택지가 너무 많거나 습관적으로 같은 메뉴만 먹음 |
| 레시피 탐색 비용 | 포털 검색 → 블로그 이동 → 재료 확인까지 평균 3~5분 소요. 정보가 분산되어 있음 |
| 외부 API 신뢰도 | 식약처 API는 비교적 안정적이나, TheMealDB는 무료 플랜으로 이미지/데이터 품질이 불균일함 |
| 반복 추천 문제 | 현재는 순수 랜덤이라 같은 메뉴가 자주 등장할 수 있음 (연속 중복은 방지하나 세션 내 중복은 미방지) |

## 타겟 사용자

**Primary (메인 타겟)**
- 직장인/자취생 20~40대, 하루 한 번 이상 "오늘 뭐 먹지?" 고민
- 음식 카테고리 취향은 있지만 구체적 메뉴 결정을 타인이나 도구에 위임하고 싶은 사람
- 요리 초보 ~ 중급자: 재료 목록과 단계별 레시피가 있으면 따라할 수 있는 수준

**Secondary (부타겟)**
- 혼밥/혼술족: 1인 기준으로 메뉴 고르기 귀찮은 사람
- 커플/소규모 그룹: "우리 오늘 뭐 먹지?" 공동 결정 도구로 공유 링크 활용

**제외 타겟**
- 식당/음식점 사장님 (배달/예약 플로우 없음)
- 전문 요리사 (고급 레시피 필요층)

## 사용자 스토리

```
AS 직장인 자취생,
I WANT 카테고리를 선택하면 빠르게 오늘 메뉴를 추천받고
        재료·레시피를 한 화면에서 확인할 수 있기를 원한다,
SO THAT 퇴근 후 30초 안에 오늘 저녁 메뉴를 결정할 수 있다.
```

```
AS 커플 중 한 명,
I WANT 추천 결과를 카카오톡으로 상대방에게 공유할 수 있기를 원한다,
SO THAT "우리 오늘 이거 만들어 먹자"는 대화를 쉽게 시작할 수 있다.
```

```
AS 요리 초보자,
I WANT 추천받은 메뉴의 재료 목록과 단계별 조리법을 볼 수 있기를 원한다,
SO THAT 마트에서 무엇을 사야 할지, 어떻게 만드는지 바로 알 수 있다.
```

### 현재 데이터 구조

- **한식/분식**: 식약처 API → KOREAN_WHITELIST 필터링 (화이트리스트 약 200개)
- **중/일/양식**: TheMealDB API → 카테고리 매핑 후 표준 UnifiedRecipe 타입으로 정규화
- **이미지**: API 제공 이미지 우선, 없으면 Pexels 검색 fallback

### 식별된 기술 부채

1. `TASTE_CHIPS` (맛 태그 8개)가 실제로 카테고리 필터 없이 `recommend(selectedCategory)` 만 호출함 - 맛 기반 필터링 로직 없음
2. `KOREAN_WHITELIST`가 하드코딩된 Set으로 유지보수 어려움 - 메뉴 추가/제거 시 코드 수정 필요
3. TheMealDB 카테고리 한국어 매핑 신뢰도 - 실제 중/일/양식 분류가 부정확할 수 있음
4. `@supabase/supabase-js` 의존성이 package.json에 있으나 현재 코드에서 사용 흔적 불명확

## 개선 방향 (To-Be)

### 핵심 방향

> **"30초 결정 → 바로 요리 시작"** 플로우를 완성한다.

현재는 메뉴 추천까지는 매끄럽지만, 추천 후 실제 요리로 이어지는 경험이 약하다.
맛 태그 필터링, 히스토리 관리, 재료 체크리스트 등으로 추천 → 실행 전환율을 높인다.

### 개선 항목

| 항목 | 현재 | 목표 |
|------|------|------|
| 맛 태그 필터링 | UI만 존재, 동작 안 함 | 각 태그와 연결된 해시태그/카테고리 매핑으로 실제 필터링 |
| 세션 내 중복 방지 | 연속 중복만 방지 | 최근 5개 히스토리 localStorage 저장, 중복 제외 |
| 재료 체크리스트 | 재료 텍스트 표시만 | 체크박스로 장보기 목록 활용 가능하게 |
| 결과 URL 공유 | 사이트 URL만 공유 | `?menu=ID` 쿼리 파라미터로 특정 메뉴 직접 공유 |
| 이미지 품질 | API 이미지 깨짐 시 Pexels fallback | 로딩 skeleton + 에러 핸들링 강화 |
| 맛 태그 데이터 연결 | 없음 | 식약처 HASH_TAG 필드 활용 |

## MVP 기능 우선순위 (P0/P1/P2)

### P0 - 즉시 수정 (버그/UX 치명 이슈)

| # | 기능 | 이유 | 예상 공수 |
|---|------|------|-----------|
| P0-1 | **맛 태그 실제 동작 구현** | 버튼이 있는데 아무 효과 없음 = UX 신뢰도 하락 | 2~3시간 |
| P0-2 | **세션 내 중복 메뉴 방지** | 같은 메뉴 반복 시 사용자 이탈 | 1~2시간 |
| P0-3 | **이미지 로딩 skeleton** | 이미지 없을 때 빈 영역이 레이아웃 깨짐처럼 보임 | 1시간 |

### P1 - 1~2주 내 개선 (핵심 가치 강화)

| # | 기능 | 이유 | 예상 공수 |
|---|------|------|-----------|
| P1-1 | **메뉴별 URL 공유** (`?menu=ID`) | 공유 시 추천 결과 재현 불가 - 카카오 공유 가치 반감 | 3~4시간 |
| P1-2 | **재료 체크리스트** | 레시피 확인 후 바로 장보기 활용 가능, 체류 시간 증가 | 2~3시간 |
| P1-3 | **추천 히스토리** (localStorage, 최근 10개) | "아까 추천받은 거 다시 보고 싶다" 수요 대응 | 3~4시간 |
| P1-4 | **다시 이 메뉴 제외 버튼** | 마음에 안 드는 메뉴 반복 제외 요청 대응 | 1시간 |

### P2 - 장기 검토 (인프라 비용 없이 가능한 것만)

| # | 기능 | 이유 | 제약 |
|---|------|------|------|
| P2-1 | **즐겨찾기** (localStorage) | 자주 먹는 메뉴 저장, 재방문 동기 생성 | 서버 없이 가능 |
| P2-2 | **오늘의 추천 고정** (날짜 기반 시드) | 같은 날 재방문 시 동일 메뉴 표시 | 서버 없이 가능 |
| P2-3 | **인원수 입력 → 재료 배수 계산** | 식약처 재료 데이터가 인분 수 미제공으로 정확도 낮음 | 데이터 한계 |
| P2-4 | **Supabase 인기 메뉴 집계** | 현재 의존성은 설치되어 있으나 미사용 - 클릭 이벤트 집계 | 무료 플랜 내 가능 |

### 제외 (현 단계에서 하지 않을 것)

- 회원가입/로그인 (인프라 비용, 복잡도 급증)
- 식단 플래너/캘린더 (1주치 계획 기능)
- 배달 앱 연동 (API 비용 발생)
- 커뮤니티/댓글 (모더레이션 리소스 필요)

| 항목 | 제외 이유 |

## 기술 제약 및 리스크

| 리스크 | 영향도 | 대응 |
|--------|--------|------|
| 식약처 API 다운타임 | 높음 | 에러 상태 UI 이미 구현됨. API 응답 로컬 캐시(sessionStorage) 추가 고려 |
| TheMealDB 무료 플랜 rate limit | 중간 | 앱 최초 로드 시 1회만 호출, useMemo로 캐싱 중 - 현재 구조로 충분 |
| Pexels API 키 노출 | 중간 | 현재 클라이언트에 직접 노출됨. Cloudflare Worker proxy로 이전 검토 |
| KOREAN_WHITELIST 하드코딩 | 낮음 | 향후 메뉴 수 확장 시 JSON 파일 분리 권장 |
| Supabase 미사용 의존성 | 낮음 | package.json에 존재하나 import 없음 - 빌드 사이즈 최적화 위해 제거 검토 |
